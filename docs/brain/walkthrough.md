# 🍵 链接池预热：首包延迟极致优化

我们已成功为 `gnet-proxy` 引入了 **精简版后端连接池**。该功能通过事先与后端握手，消除了流量到来时的 TCP 建立开销。

## 🌟 新增核心能力

- **事先预热 (JumpStart)**: 程序启动或配置加载后，会自动并发拨号至目标后端，建立“待命”链接。
- **健康探测 (Health Check)**: 连接池在交出链接前，会进行一次 `Read` 级别的极速健康检查，确保链接不是“僵尸链接”。
- **空闲超时 (IdleTimeout)**: 支持为每个路由独立配置超时时间，自动清理池子中长期不用的陈旧链接，保护后端资源。
- **自动补货**: 后台协程会实时监控池水位，一旦链接被取用，立即启动异步拨号补充。

## 📄 配置示例 (config.jsonc)

您现在可以在配置文件中为特定路由开启预热：

```jsonc
{
  "routes": {
    "updates.cdn-apple.com": {
      "addr": "tcp://127.0.0.1:10443",
      "jump_start": 4,      // 💡 保持 4 个空闲链接待命
      "idle_timeout": 300   // 💡 5 分钟不使用则自动销毁
    }
  }
}
```

## 🛠️ 内部实现精要

1. **`pkg/outbound/pool.go`**: 采用分片 Channel 锁设计，确保在高并发下获取链接的锁竞争极低。
2. **`replenishLoop`**: 采用 Ticker 驱动的异步补货逻辑，平衡了补货速度与 CPU 损耗。
3. **`Acquire` 逻辑**: 实现了优雅的降级机制——如果池子瞬间被取空，会自动回退到“现场拨号”模式，保证业务不中断。
