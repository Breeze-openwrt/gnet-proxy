# 🏗️ gnet-proxy 深度架构分析 (Linux 工业级)

这份文档深入探讨了 `gnet-proxy` 的设计理念、高性能背后的逻辑以及全链路流控模型。

---

## 1. 核心架构模式：DDD (领域驱动设计)
本项目将业务逻辑拆分为三个核心层级，实现了逻辑的完全解耦：

### 🧱 领域模型 (Domain Model)
- **配置定义** (`pkg/config`): 业务规则的静态描述。
- **路由核心** (`pkg/core`): 决策系统的灵魂，不依赖任何网络框架。

### 🏗️ 基础设施 (Infrastructure)
- **gnet 引擎**: 基于 Reactor 模型的高效 I/O 多路复用。
- **内存复用池** (`pkg/common/pool`): 32KB/64KB 动态配比字节切片复用，极大减轻 GC 负担。

### 🚀 应用服务 (Application Service)
- **Server** (`pkg/inbound`): 协调者，负责将流量嗅探、路由决策与转发链路串联。

---

## 2. 流量控制：全链路智能背压 (Bidirectional Backpressure)

为了在极速下载与大文件上传之间取得平衡，系统实现了“双流向”闭环控制：

### 📤 上行流控 (Upload: 64MB)
- **缓冲层**：2048 个积压插槽 (约 64MB)。
- **高水位 (80%)**：当客户端发货太快，后端消费不及时导致积压超过 1600 个包时，进入“读拦截”模式，迫使客户端 TCP 窗口收缩。
- **低水位 (20%)**：降至 400 个包以下时，通过 `gnet.Conn.Wake()` 唤醒事件循环，恢复全速接收。

### 📥 下行流控 (Download: 8MB)
- **发送延迟感应**：利用 `OutboundBuffered()` 监控 `gnet` 内部发送队列。
- **背压阀门**：如果由于客户端接收慢导致数据积压超过 **8MB**，代理会主动挂起后端物理连接的读取动作，保护内存不产生溢出。

---

## 3. 连接健壮性：铁律级保活策略

- **高频探测**：TCP Keep-Alive 周期缩短至 **30s**，对抗复杂网络环境下的静默断路。
- **双向死亡截止时间 (Deadlines)**：
  - 上行写保护：5 分钟。超过此时间后端不收包，强制断开。
  - 下行读保护：5 分钟。超过此时间后端不回包，强制断开。
- **二次必启 (Safe Reuse)**：
  - 标准 PID 路径：`/run/gnet-proxy.pid`。
  - 重启自清理：新进程启动前会自动清理残留 PID 对应的僵尸进程，确保 443/TCP 端口始终可以被最新实例占用。

---

## 4. 总结：设计者的初衷
`gnet-proxy` 专注于 **高性能、工业级稳定性、零运维焦虑**。它通过透明转发与精准的背压反馈，在有限的硬件资源下实现了支撑 TB 级流量的吞吐潜力。
